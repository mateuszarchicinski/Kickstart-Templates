Angular v1.x Szablon

Szablon udostępnia skonfigurowane i zautomatyzowane środowisko pracy z możliwością dalszej rozbudowy. Wykorzystuje do tego takie technologie jak NodeJS, NPM, Bower, GulpJS i Karma. W podstawowej konfiguracji jest przygotowany do rozwoju aplikacji webowej po stronie klienta w technologii Angular v1.x - MVC Framework.

Przygotowany szablon aplikacji webowej Angular v1.x udostępnia w pełni skonfigurowane i zautomatyzowane środowisko pracy za pomocą takich technologii jak NodeJS, NPM, Bower, GulpJS i Karma.

Użycie

Projekt Angular 1.x Szablon jest dostępny wyłącznie na Github. Przykłady instalacji poniżej. Przy dwóch pierwszych wymagany jest system kontroli wersji Git.

  1. Za pomocą konsoli utwórz nowy katalog i przejdź do niego: 
    mkdir katalog-projektu && cd katalog-projektu
    W nowym katalogu zainicjuj środowisko komendą git init i następnie wywołaj: 
    git pull repozytoriumhttps://github.com/mateuszarchicinski/Angular1.x-Template
  2. W konsoli wywołaj polecenie sklonowania całego repozytorium, które tym samym utworzy nowy katalog: 
    git clone repozytoriumhttps://github.com/mateuszarchicinski/Angular1.x-Template
  3. Manualnie pobierz najnowszą stabilną wersje z strony publikacji szablonu.

Po wykonaniu dowolnej instalacji. Przejdź do katalogu projektu i wywołaj polecenia npm install i bower install.

W tym momencie można zacząć korzystać z wszystkich zdefiniowanych zadań w gulpfile.js. Tabela zadań dostępna tutaj.

Konfiguracja

Globalne ustawienia są dostępne w pliku project.config.js, który umożliwia zmianę:

  • nazwy aplikacji,
  • języków,
  • nazwy plików konfiguracyjnych i danych,
  • katalogów pracy i produkcji (wymagana również manualna zmiana w pliku .bowerrc),
  • dostępów FTP,
  • podania kluczy do zewnętrznych API.

Dodatkowo może posłużyć nam do przechowywania różnego rodzaju zmiennych z których będziemy korzystać indywidualnie. W pliku gulpfile.js odwołujemy się do niego poprzez zmienną PROJECT_CONFIG.

Gulp zadania

Zadania uruchamiamy z poziomu konsoli będąc jednocześnie w głównym katalogu projektu wpisując słowo kluczowe gulp w połączeniu z nazwą i opcjonalnymi argumentami zadania do wykonania.

gulp nazwa opcjonalne argumenty

Przykładowe polecenie będzie wyglądać następująco gulp pug --lang pl (nr: 6).

Poniższa tabela zawiera informację dotyczące nazwy, opcjonalnych argumentów i opisu do każdego zdefiniowanego zadania.

Nazwa Opis

sass:css
  1. Wykorzystuje plik main.scss lub main.sass.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Inicjuje utworzenie Source Map na podstawie głównego pliku Sass (gulp-sourcemaps).
  4. Kompiluje synchronicznie kod Sass na kod Css o stylu wyjściowym nested (gulp-sass).
  5. Dodaje właściwe przedrostki z metodami do kodu Css. Uwzględniając przy tym wsparcia dla wskazanych przeglądarek w obiekcie konfiguracyjnym (gulp-autoprefixer).
  6. Zapisuje Source Map do pliku main.css.map.
  7. Zapisuje uzyskany kod Css do pliku main.css.
  8. Wstrzykuje nowy kod Css do strony podglądowej (browser-sync).

sass:lint
  1. Wykorzystuje pliki z rozszerzeniem .scss lub .sass.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Sprawdza kod Sass pod kątem wybranych reguł
    w obiekcie konfiguracyjnym (gulp-sass-lint).
  4. Wyświetla odpowiednio sformatowane komunikaty
    o wykrytych niezgodnościach w kodzie Sass (Formatter).

js:hint
  1. Wykorzystuje pliki z rozszerzeniem .js.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Sprawdza kod Js pod kątem błędów i optymalizacji (gulp-jshint).
  4. Wyświetla odpowiednio sformatowane komunikaty
    o wykrytych błędach i możliwej optymalizacji w kodzie Js (Reporter).

js:test
  1. Uruchomi testy jednostkowe w środowisku Karma, które konfigurujemy za pomocą pliku karma.conf.js. Wymieniony plik umożliwia wybór frameworków testujących i wielu innych rzeczy o których można poczytać tutaj.

jade:pug
  1. Wykorzystuje pliki z rozszerzeniem .jade.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Zamienia rozszerzenie plików .jade na .pug (gulp-rename).
  4. Zapisuje pliki z nowym rozszerzeniem .pug
    w analogicznych lokalizacjach do plików .jade.
  5. Usuwa nie potrzebne pliki z rozszerzeniem .jade (del).

pug
--lang pl / en
  1. Sprawdza czy przekazano dodatkowe argumenty do procesu zadania. Ustawia początkową wartość zmiennej langValue na "pl" i nadpisuje ją w przypadku podania innej.
  2. Wykorzystuje pliki z rozszerzeniem.pug.
  3. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  4. Przygotowuje obiekt danych przekazując go wewnątrz strumienia do gulp-pug (gulp-data).
  5. Kompiluje pliki .pug zwracając uzupełniony kod Html
    o przekazane informacje w obiekcie danych (gulp-pug).
  6. Wstrzykuje tagi Link i Script z ścieżkami źródłowymi do zainstalowanych bibliotek Css lub Js. Pomiędzy poniższe komentarze w kodzie Html (wiredep): 
    <!--bower:css-->
    // <link href="">
    <!--endbower-->

    <!--bower:js-->
    // <script src="">
    <!--endbower-->
  7. Zapisuje uzyskane kody Html do index.html i plików widoku.

pug:lint
  1. Wykorzystuje pliki z rozszerzeniem .pug.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Sprawdza kod Pug pod kątem wybranych reguł w obiekcie konfiguracyjnym (gulp-pug-lint).
  4. Wyświetla odpowiednio sformatowane komunikaty
    o wykrytych niezgodnościach w kodzie Pug.

html
  1. Wykorzystuje pliki z rozszerzeniem .html.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Filtruje kod Html pod kątem wstawionych komentarzy: 
    <!--build:css main.min.css-->
    // <link href="">
    <!--endbuild-->

    <!--build:js main.min.js-->
    // <script src="">
    <!--endbuild-->
    Zastępując je pojedynczymi tagami Link i Script odnoszącymi się do plików źródłowych main.min.css
    i main.min.js powstałymi w skutek połączenia wszystkich załączonych bibliotek Css lub Js znajdujących się pomiędzy tymi komentarzami (gulp-useref).
  4. Sprawdza typ przetwarzanych plików w strumieniu. Wywołując odpowiednie metody minifikujące dla plików
    z rozszerzeniem .css i .js (gulp-clean-css / gulp-uglify).
  5. Zapisuje uzyskane kody Css i Js do plików w katalogu produkcyjnym.

html:hint
  1. Wykorzystuje pliki z rozszerzeniem .html.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Sprawdza poprawność wygenerowanego kodu Html według reguł wskazanych w obiekcie konfiguracyjnym (gulp-htmlhint).
  4. Wyświetla odpowiednio sformatowane komunikaty
    o wykrytych niezgodnościach w kodzie Html (Reporter).

html:minify
  1. Wykorzystuje pliki z rozszerzeniem .html.
  2. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  3. Minifikuje kod Html zgodnie z ustawieniami przekazanymi w obiekcie konfiguracyjnym (gulp-htmlmin).
  4. Zapisuje uzyskane kody Html w tych samych lokalizacjach plików źródłowych na których rozpoczęto wykonywanie zadania (nadpisywanie).

server

  1. Uruchamia lokalnie serwer nasłuchujący na katalog pracy pod adresem http://localhost:3000/ (widoczny w konsoli).

    Udostępniając tym samym metody pomocnicze, które umożliwiają odświeżenie przeglądarki np. w momencie wprowadzenia zmiany w dowolnym pliku lub dynamiczne wstrzyknięcie styli Css bez przeładowania strony podglądowej.

    Zapewnia również podgląd aplikacji z każdego urządzenia podłączonego do sieci lokalnej pod adresem zewnętrznym (External - widoczny w konsoli).

    Więcej informacji na temat konfiguracji i możliwości moduł Browser Sync tutaj.

watch

  1. W następstwie wprowadzanych zmian w plikach wywołuję odpowiednie zadania:

    PlikiZadania
    .sass lub .scsssass:lint2 i sass:css1
    .jsjs:hint3 i browserSync.reload11
    project.data.json i .pugpug:lint7 i pug6
    .htmlhtml:hint9 i browserSync.reload11

clean
  1. Usuwa katalog produkcyjny wraz z całą zawartością. Uruchamiane zawsze na początku tworzenia nowego builda aplikacji (del).

copy
  1. Kopiuje niżej wskazane tzw. pliki statyczne z katalogu pracy do katalogu produkcyjnego:
    • /files/*
    • /fonts/*
    • /img/*
    • /*.png
    • /*.xml
    • /*.ico

images
--option advanced
  1. Sprawdza czy przekazano dodatkowe argumenty do procesu zadania. Ustawia tym samym rozszerzenia plików do optymalizacji, moduł optymalizujący i argumenty dla wybranego modułu.
  2. Zarówno przy opcji standardowej (bez dodatkowych argumentów) i zaawansowanej (z dodatkowymi argumentami) wykorzystane zostaną grafiki
    z rozszerzeniem .jpg i .png. W standardowej opcji dochodzą jeszcze dwa typy .svg i .gif.
  3. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  4. Standardowa optymalizacja grafik odbywa się za pomocą modułu gulp-imagemin.
  5. Zaawansowana optymalizacja grafik odbywa sie za pomocą modułu gulp-tinify wykorzystującego TinyPNG API.
  6. Zapisuje zoptymalizowane grafiki w tych samych lokalizacjach grafik źródłowych na których rozpoczęto wykonywanie zadania (nadpisywanie).

images:sprite
--name _sprite
  1. Sprawdza czy przekazano dodatkowe argumenty do procesu zadania. Ustawia początkową wartość zmiennej spriteCssName na "_sprite" i nadpisuje ją w przypadku podania innej.
  2. Wykorzystuje grafiki z rozszerzeniem .jpg, .png i .gif.
  3. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  4. Łączy wszystkie grafiki z katalogu img > sprites_sources w jedną grafikę złożoną z wielu. Zwracając tym samym automatycznie wygenerowany plik .scss zawierający gotowe mixiny do użycia, umożliwiające wyświetlenie dowolnej grafiki wcześniej złączonej (gulp-spritesmith).
  5. Sprawdza rozszerzenia przetwarzanych plików
    w strumieniu (gulp-if). Zapisując odpowiednio pliki
    z rozszerzeniem .png w katalogu img i .scss w katalogu sass > components > sprites.

upload
--upload
  1. Sprawdza czy wszystkie wartości zmiennych dotyczących połączenia FTP zostały zdefiniowane w pliku project.config.js i czy przekazano dodatkowy argument do procesu zadania.
  2. Wykorzystuje wszystkie pliki z katalogu produkcyjnego.
  3. Raportuje o napotkanych błędach w strumieniu nie przerywając wykonywania kolejnych pipe'ów (gulp-plumber).
  4. Przesyła pliki na serwer FTP przekazując lokalizację zapisu poprzez zmienną DESTINATION (vinyl-ftp).

    i przyśpieszą proces tworzenia nowej aplikacji.

Więcej informacji i kod źródłowy do sprawdzenia w pliku gulpfile.js.

Testowanie

Uruchamia się za pomocą zadania gulp js:test. Do obsługi testów jednostkowych, nasłuchu zamian w plikach źródłowych i raportowania bledów zostało przygotowane środowisko Karma. Standardowa konfiguracja wykorzystuje framework Jasmine, symulator PhantomJS i reporter Spec. Pliki konfiguracji i testów znajdują się Folder projektu > Test > Karma.conf.js / Spec.
